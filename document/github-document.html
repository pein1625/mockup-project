Repo

Repository là kho chứa source code và những thông tin để duy trì, quản lý các sửa đổi, lịch sử của project.

Có 2 loại repository:
1. Remote repository
2. Local repository

Remote repository

Là repo được bố trí trên git server, có thể chia sẻ cho thành viên team hay nhiều người khác.

Các git server thông dụng hiện nay:
- github
- bitbucket
- gitlab

Cách tạo remote repository:
1. Lên git server
2. Tìm tới nút “New repository” và click
3. Điền repository name, description(có thể bỏ qua), chọn chế độ public/private, chọn khởi tạo file readme, .gitignore, liscense(có thẻ bỏ qua),

Local repository

Là repo được bố trí trên máy tính cá nhân, dành cho một người sử dụng hay dành cho những user của máy tính đó sử dụng.

Tạo local repository:

  • Cách 1:
    1. Mở terminal, cd đến thư mục muốn tạo repository
    2. Dùng lệnh git init repository_name để tạo 1 folder repository có tên repository_name.
  • Cách 2:
    1. Mở terminal, cd đến thư mục muốn tạo repository
    2. mkdir repository_name tạo folder mới
    3. cd repository_name vào folder vừa tạo
    4. Dùng lệnh git init để biến folder hiện tại thành một repository
  • Cách 3:
    1. Tạo một remote repository (hay tìm một remote repository đã có sẵn)
    2. Copy remote_repo_url của remote repository đó
    3. Mở terminal lên và cd đến thư mục muốn tạo repository
    4. Dùng lệnh git clone remote_repo_url để thực hiện clone remote repository vừa rồi.
    5. Khi này, local-repo bạn vừa tạo sẽ có tên giống với tên remote-repo, nếu muốn dùng một tên repo khác, dùng lệnh git clone remote_repo_url other_name

Branch

Branch là là đối tượng quan trọng, là thế mạnh của git và hầu hết các version control system.
Branch dùng để phân nhánh trong quá trình phát triển mà không làm ảnh hưởng đến nhánh chính.
Khi branch hoàn thành một công việc nào đó, ta có thể gộp branch đó vào nhánh chính.

Các thao tác cơ bản với branch:

Truy cập branch:

  • git checkout branch_name

Tạo mới branch:

  • Cách 1: git branch branch_name -> tạo mới một branch là branch_name, lệnh này sẽ báo lỗi fatal nếu branch_name đã tồn tại
  • Cách 2: git checkout -b branch_name -> tạo mới và truy cập vào branch_name, lệnh này sẽ báo lỗi fatal nếu branch_name đã tồn tại

Xoá branch:

  • Cách 1: git branch -d branch_name -> dùng để xoá những branch đã được merge vào nhánh chính (–delete)
  • Cách 2: git branch -D branch_name -> dùng để xoá những branch bất kể đã được merge vào nhánh chính hay chưa (–delete –force)

Merge branch:

  • git checkout branch_b -> truy cập branch b
  • git merge branch_a -> merge branch a vào branch b

Rebase

  • git checkout branch_b
  • git rebase branch_a -> rebase branch_b với branch_a

Đổi tên branch:

  • git branch -m new_branch_name -> đổi tên branch hiện tại thành new_branch_name
  • git branch -m old_branch_name new_branch_name -> đổi tên branch old_branch_name thành new_branch_name khi đang ở branch bất kỳ.

Tags

git tag dùng để đánh dấu các mốc quan trọng của dự án - thường được dùng để dánh dấu các thời điểm phát hành (v1.0)

Các thao tác với tags

Liệt kê tag

Để liệt kê tag, chỉ cần sử dụng lệnh git tag. Lệnh này sẽ liệt kê các tag theo thứ tự bảng chữ cái

$ git tag
v1.0
v1.1

Thêm tag mới

git sử dụng 2 loại tag chính:
1. lightweight tag(hạng nhẹ): dùng để trỏ đến một commit nào đó
2. annotated tag(chú thích): được lưu trữ như là những đối tượng đầy đủ trong cơ sở dữ liệu của Git, chứa tên người tag, địa chỉ email và ngày tháng; có thông điệp kèm theo; và có thể được ký và xác thực bằng GNU Privacy Guard (GPG).

Thông thường, annotated tag được khuyến khích sử dụng hơn; tuy nhiên muốn một tag tạm thời không lưu trữ thông tin, lightweight tag là sự lựa chọn hợp lý.

Annotated tag

Tạo annotated tag
- git tag -a tag_name -m "message"
- tham số -a để khai báo tạo annotated tag
- tham số -m để truyền vào nội dung/thông điệp cho tag. Nếu không chỉ định nội dung, Git sẽ mở trình soạn thảo và yêu cầu bạn nhập nội dung vào đó.
Xem thông tin tag và commit:
- git show

Lightweight tag
  • git tag tag_name -> tạo lightweight tag cho commit gần nhất
  • git show tag_name -> nếu tag_name là lightweight tag, lệnh này chỉ show commit nó gắn vào.

Issue

Issue là nơi theo dõi các task, các cải tiến hay bug của dự án. Issue có thể được chia sẻ, bàn luận giữa các thành viên team.

  • Một issue có phần title và description mô tả issue đó
  • Phần label dùng để phân loại issue
  • Phần assignee để phân định cho một thành viên chịu trách nhiệm giải quyết toàn bộ issue.
  • Phần comments để mọi người có thể vào bình luận, phản hồi ý kiến.

Wiki

Wiki là phần lưu trữ những tài liệu mô tả dự án, giúp cho các thành viên hay những người khác dễ dàng hiểu hơn về dự án như là dự án này là gì, được thiết kế như thế nào, có những quy tắc gì,.. từ đó có thể nhanh chóng tham gia đóng góp và mở rộng dự án hơn.

  • Wiki được xem như là tài liệu bổ xung, đầy đủ hơn cho file readme.md.

Managing (project)

Bảng project giúp tổ chức, phân chia độ ưu tiên và cập nhật trạng thái công việc.

Template for project managing

  • Từ mục project, ta có thể tạo ra các cột To Do, In Progress, Done (Resolved).
  • Trong cột To Do chứa các task được xắp xếp theo độ ưu tiên, mỗi task được mô tả chi tiết.
  • Khi một thành viên nhận task, họ sẽ kéo task đó sang cột In Progress, thay đổi trạng thái task.
  • Khi task được hoàn thành, nó sẽ được kéo sang cột Done (resolved).

Securing

Coding secure

  • Một trong những mục đích của git là bảo vệ code, tránh khỏi các nhầm lẫn, sai lầm khi phải copy, paste code qua lại giữa các thành viên team. Các branch được tạo ra để sửa code mà không ảnh hưởng đến những branch khác. Hay các commit được ghi lại ở các thời điểm khác nhau, lưu lại lịch sử coding. Khi xảy lỗi hoặc có vấn đề gì đó về code, ta có thể trở lại commit trước đó và làm lại một cách đễ dàng. Khi có một issue, chúng ta đơn giản chỉ cần tạo một branch mới để giải quyết, và code này sẽ được merge khi issue được giải quyết xong.

Git server secure

  • Các git server đều có chế độ public/private cho các repository.
  • Ở chế độ public, tất cả mọi người đều có thể view, fork source code, cũng có thể trở thành contributer của project.
  • Ở chế độ private, chỉ có chủ sở hữu repository và những thành viên được thêm mới có thể thực hiện những thao tác trên.
  • User có thể dễ dàng tạo một public repository trên các git server. Tuy nhiên, để tạo một private repository, user có thể phải trả tiền cho git server để duy trì chế độ private(github). Hoặc là chỉ những repository nhất định mới có thể private (<= 5 thành viên tham gia repo - bitbucket), và trả tiền cho những trường hợp còn lại.

Git flow secure

  • Trong quá trình thực hiện project, các lỗi như push, push -f nhầm branch của người khác, xoá nhầm remote branch, push nhầm branch master sẽ không còn nếu như các thành viên team thực hiện fork repository về tài khoản của mình thay vì dùng chung một repo. Khi này repo chính sẽ chỉ để một người quản lý, các thành viên sẽ chỉ push branch lên repo mình fork về.

Fork

  • Việc “fork” cũng giống như branching vậy. Nếu như git tạo ra các branch để làm việc, sửa đổi code mà không ảnh hưởng tới branch master, thì fork cũng thế. Từ một repository được tạo ra trước đó, các thành viên fork về tài khoản của mình và làm việc. Họ có thể thêm xoá chỉnh sửa tuỳ ý trên fork repo của mình mà không làm ảnh hưởng tới repo chính.
  • Khi một thành viên hoàn thành công việc nào đó trên fork repo của mình, họ có thể gửi pull request đến một branch nào đó của repo gốc.
  • Bạn không thể push lên repo của người khác mà không có sự cho phép ( không phải là contributor ), nhưng có thể fork repo về, sửa đổi code rồi gửi pull request.
  • Cách fork một repo trên github:
    1. Trên github, truy cập vào một repository để fork
    2. Từ góc trên bên phải, click vào chữ “fork”
